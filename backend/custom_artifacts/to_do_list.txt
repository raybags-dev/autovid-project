We currently define prompt lists in code:

- DEFAULT_SHORT_TOPICS
- DEFAULT_PROMPTS

These are used by an auto-generation pipeline. The issue is that prompts are randomly selected, which causes repeated selections and results in duplicate or highly similar generated videos.

Additionally, we have a stock footage pipeline (using the Pexels API), but it has the following issues:
- It frequently reuses the same footage clips
- It sometimes fails to retrieve enough footage to cover the full script duration
- Footage selection is not well aligned with script segments

We also have a UI issue:
- The option to use footage as a background is no longer clearly available across all MP4 generation pipelines
- If it exists, it is unclear or inconsistently exposed
- This option should be clearly available and consistently named **"Stock Footage"**

---

## Requirements

### 1. Prompt Uniqueness System
Design a deterministic system that ensures:

- No prompt is reused until all prompts are exhausted
- State persists across runs
- Supports:
  a) finite list cycling (no repetition)
  b) dynamic prompt generation when exhausted

If using Supabase, define:
- schema for prompts
- tracking usage (used, unused, timestamps)
- safe reset or expansion strategy

Also include:
- semantic deduplication (avoid similar prompts, not just exact matches)

---

### 2. Stock Footage Pipeline Fix

Redesign the stock footage pipeline to:

- Ensure **diversity of footage** (avoid reusing the same clips)
- Retrieve **sufficient footage to match script duration**
- Improve **semantic matching** between script segments and footage

The solution must include:

#### a) Footage selection strategy
- Break script into segments (per sentence or time block)
- Generate search queries per segment
- Fetch multiple candidate clips per query
- Rank and select best matches

#### b) De-duplication
- Track previously used video IDs
- Prevent reuse within:
  - same video
  - recent history (configurable window)

#### c) Coverage enforcement
- Ensure total footage duration >= narration duration
- If insufficient footage:
  - fallback strategies (reuse with variation, extend clips, or fetch broader queries)

#### d) Variation strategy
- Use multiple queries per segment (synonyms, paraphrases)
- Avoid repeating identical search terms

### 1.1 Prompt Exhaustion Handling (MANDATORY)

When all prompts in a list are exhausted, the system must support TWO explicit modes:

#### Mode A — Automatic Prompt Generation
- The system should generate a new batch of prompts automatically
- These prompts must follow a consistent thematic direction based on:

  → dark, philosophical, and existential themes  
  → including (but not limited to): meaning of life, death, love, absurdism, isolation, purpose, and human condition  

- Generated prompts must:
  - be semantically unique from previous prompts
  - avoid repetition in structure and wording
  - maintain variety while staying within the theme

#### Mode B — Manual Input
- The user must be able to:
  - upload a new list of prompts (file input), OR
  - enter prompts as a comma-separated string:

    Example:
    "prompt 1, prompt 2, prompt 3"

- The system must:
  - validate and parse input
  - deduplicate against existing prompts
  - store them in the database for future use

#### Mode Selection
- The system should allow configuration to:
  - automatically choose Mode A, OR
  - require user input (Mode B), OR
  - support a hybrid fallback:
    → try auto-generation, fallback to manual if generation fails or is disabled
